const handleImportPendingSessions = () => {
  const pending = patient?.pendingSessions || patient?.balanceSessions || [];

  const mappedSessions = pending.map((session) => ({

## 🎯 Como conectar o Pacote nesse fluxo (sem criar nada novo)

### Opção A: Apenas visualização/importação no front (mais rápida)

O Pacote consome `GET /api/v2/balance/:patientId` e mapeia `v2_financial.items` para sessões:

```ts
const { data } = await api.get(`/v2/balance/${patientId}`);
const pendingItems = data.v2_financial.items;

const sessionsToImport = pendingItems.map(item => ({
  id: item._id,                    // ID do Payment (source of truth)
  amount: item.amount,
  date: item.serviceDate || item.paymentDate,
  time: item.appointment?.time,
  specialty: item.specialty,
  doctorId: item.doctor?._id,
  doctorName: item.doctor?.fullName,
  source: "BALANCE_IMPORTED",      // marca pra não duplicar
  status: "PENDING"
}));
```

E injeta no estado do pacote:
```ts
setPackageSessions(prev => [...prev, ...sessionsToImport]);
```

> ⚠️ Aqui o Pacote **não quita** os débitos no backend. Ele só "visualiza/importa" como sessões pendentes.

---

### Opção B: Pacote quita os débitos no backend (mais completa)

Se o Pacote deve **quitar** os débitos (marcar como pagos/settled), aí precisa de persistência.

Hoje o `POST /api/v2/packages` já faz isso na criação via `selectedDebts` (que usa `PatientBalance` legado).

Para um Pacote **existente**, precisaríamos de um novo endpoint:

```
POST /api/v2/packages/:packageId/attach-debts
```

Body:
```json
{
  "debtIds": ["payment_id_1", "payment_id_2"]
}
```

O que ele faria:
1. Busca os `Payment` pelos IDs
2. Cria sessões no pacote com `source: "BALANCE_IMPORTED"`
3. Atualiza os `Payment` para `status: "settled"` e `settledByPackageId: packageId`
4. Retorna as sessões criadas

---

## 🏁 Minha recomendação

**Comece com a Opção A** (frontend-only usando `GET /api/v2/balance/:patientId`):

- ✅ Não precisa criar endpoint novo
- ✅ Não duplica regra financeira
- ✅ Não recalcula nada
- ✅ O Pacote "liga" no flow existente da v2
- ✅ Source of truth continua sendo `Payment` (v2)

Se depois vocês precisarem que o Pacote **quite** os débitos (marcar como pagos no backend), aí criamos o `POST /api/v2/packages/:packageId/attach-debts`.

---

## ⚠️ Atenção: inconsistência que existe hoje

O `POST /api/v2/packages` (criação) usa `PatientBalance` (legado) para quitar débitos via `selectedDebts`. Mas a **fonte de verdade V2** é o `Payment`.

Na prática:
- `GET /api/v2/balance/:patientId` → retorna `Payment` (V2, correto)
- `POST /api/v2/packages` → espera IDs do `PatientBalance.transactions` (legado)

Isso significa que os `debtIds` que vêm da v2 (`Payment._id`) **não são os mesmos IDs** que o `POST /api/v2/packages` espera (`PatientBalance.transactions._id`).
